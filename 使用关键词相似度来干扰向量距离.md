## 节点距离二因子加权融合方案

### 问题

我们的项目中因果星空中的事件视界距离即向量距离，但是，包含了相同词句的两个节点却会相距很远的情况，为些我们考虑以下算法
1，向量距离（目前的实现），语义越相似距离越短
2，关键字相似度（目前引用的v5-relevance包，有1对N和1：1的相似度）相似度越高就距离越短

那么二者如何权重，我想1是最主要的，用2来干扰

### 核心思路

将 `get_event_horizon` 从"纯 SQL 向量距离"改为"Python 端二因子加权综合距离"：

```
D_final = W1 × D_vector + W2 × D_keyword
```

默认权重（可配置）：**W1=0.6（向量）> W2=0.4（关键字）> **

### 三种距离的归一化设计

| 因子 | 计算方式 | 范围 | 转换为距离 |
|------|----------|------|-----------|
| **D_vector**（主要） | `(semantic_vector <=> target_vector) * 100`（现有SQL） | 0-100 | 直接使用 |
| **D_keyword**（次要） | `v5.calculate_relevance_score(候选节点.event_tuple, 焦点节点.event_tuple)` | 0-130+ | `max(0, 100 - min(v5_score, 100))` → 0-100 |


这样综合距离范围仍为 **0-100**，与现有 `max_eyes` 阈值完全兼容。

### 实现步骤

#### 1. 新增辅助方法 `_compute_causal_hop_distances`（`core/database.py`）
- 从焦点节点出发，沿 `parent_id`（向上）和子节点关系（向下）做 BFS
- 返回 `{node_id: hop_count}` 字典
- 本owner、actor下全息遍历

#### 2. 重构 `get_event_horizon`（`core/database.py`）
- **第一阶段（SQL）**：获取所有同 owner、actor 且有 semantic_vector 的节点，计算向量距离作为候选集（不再用 `max_eyes` 过滤，而是获取全部候选）
- **第二阶段（Python）**：
  - 延迟导入 `v5`（避免循环依赖）
  - 对每个候选节点计算 `D_keyword`（v5 分数）
  - 计算综合距离 `D_final = W1×D_vector + W2×D_keyword`
- **第三阶段**：按 `D_final <= max_eyes` 过滤，按 `D_final` 升序排序
- 返回结果中增加 `distance_vector`、`distance_keyword`字段，方便调试

#### 3. 配置项（`.env`）
新增以下配置（均有默认值，无需修改即可工作）：
```
DISTANCE_WEIGHT_VECTOR=0.6
DISTANCE_WEIGHT_KEYWORD=0.4
CAUSAL_HOP_DISTANCE=30
```

#### 4. 前端无需改动
前端力导向布局使用的是 `link.similarity`（语义连线），不直接使用事件视界的 `distance` 字段。事件视界仅决定哪些节点被标记为"视界内节点"。

### 关键设计决策

1. **v5 调用方向**：`v5.calculate_relevance_score(候选节点.event_tuple, 焦点节点.event_tuple)` — 焦点节点作为 query，候选节点作为 content。这样焦点节点中的关键词如果在候选节点中出现，就会得高分，正好解决"相同词句距离远"的问题。

2. **延迟导入 v5**：在方法内部 `from v5 import V5Relev`，避免 `database.py` ↔ `search.py` 循环依赖。v5 实例可缓存为类属性，避免重复初始化 jieba 词典。

3. **候选集策略**：获取所有同 owner、actor 节点而非按向量距离预筛，确保高 v5 分数但向量距离远的节点不被遗漏。

4. **性能考量**：v5 基于 jieba 分词，单次计算约 1-5ms；BFS 限制 5 跳。对于百级节点，总计算量可接受。

### 影响范围
- `core/database.py`：修改 `get_event_horizon`，新增 `_compute_causal_hop_distances`
- `.env`：新增 5 个配置项（可选）
- `main.py`：无需改动（调用接口不变）
- `static/js/3d_main.js`：无需改动

### 风险
- **性能**：节点数量大时（>500），v5 计算可能较慢。但后期数据库在ramdisk上，可以不用考虑。
- **v5 分数不对称**：A对B和B对A的v5分数可能不同。当前使用单向计算，如效果不佳可改为双向取最大值。
