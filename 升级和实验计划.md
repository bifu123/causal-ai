# 因果AI 3D可视化升级和实验计划

## 项目概述

基于对《因果星空（三）3D 视觉实现蓝图.md》的评估，当前项目使用Vis-Network实现2D因果可视化，存在因果链条多时显示性能问题。为实现蓝图中描述的"因果星空"愿景，计划将前端可视化从Vis-Network迁移到Three.js + 3d-force-graph，实现3D沉浸式因果可视化。

## 一、当前状态评估

### 1.1 技术架构现状
- **前端**：Vis-Network (2D) + Socket.IO + Tailwind CSS
- **后端**：FastAPI + PostgreSQL + 代谢引擎
- **数据流**：WebSocket实时同步
- **核心功能**：因果节点CRUD、多父节点支持、图片管理、代谢衰减

### 1.2 与蓝图对比差距

| 蓝图阶段 | 蓝图要求 | 当前状态 | 差距分析 |
|---------|---------|---------|---------|
| **阶段一：逻辑建模期** | Three.js + 3d-force-graph<br>相机巡航<br>物理模拟 | Vis-Network (2D)<br>无相机动画<br>无3D物理 | 技术栈不匹配，缺乏3D能力 |
| **阶段二：感官震撼期** | 粒子系统、GPU计算<br>辉光效果、空间扭曲 | 基础代谢引擎<br>无视觉特效 | 缺乏高级视觉效果 |
| **阶段三：沉浸体验期** | Unity/Unreal引擎<br>流体动力学、空间音效 | 基础2D交互<br>无音效 | 架构级差距 |

### 1.3 主要问题
1. **性能问题**：节点数量多时Vis-Network性能下降
2. **可视化限制**：2D难以展示复杂因果层次
3. **体验不足**：缺乏沉浸式3D体验
4. **扩展困难**：难以实现蓝图中的高级特效

## 二、升级目标

### 2.1 总体目标
将前端可视化从Vis-Network迁移到Three.js + 3d-force-graph，实现基础的3D因果可视化，解决大规模节点显示问题，为后续高级特效奠定基础。

### 2.2 阶段目标
1. **短期（4-6周）**：完成基础3D可视化迁移
2. **中期（8-12周）**：实现蓝图阶段一核心功能
3. **长期（16-24周）**：逐步实现阶段二、三高级特效

### 2.3 成功标准
- 3D场景正常渲染，性能优于2D版本
- 支持1000+节点流畅显示
- 保持现有所有功能
- 用户交互体验提升

## 三、技术方案

### 3.1 新前端技术栈
- **Three.js**：3D渲染引擎
- **3d-force-graph**：3D力导向图库（基于d3-force-3d）
- **Socket.IO客户端**：保持现有实时通信
- **Tween.js**：相机动画控制
- **自定义Shader**：未来扩展粒子效果

### 3.2 架构设计
```
新前端架构：
┌─────────────────────────────────────────┐
│          用户界面层 (UI Layer)           │
│  ┌─────────────┐  ┌─────────────┐      │
│  │  抽屉详情   │  │  控制面板   │      │
│  └─────────────┘  └─────────────┘      │
└─────────────────────────────────────────┘
┌─────────────────────────────────────────┐
│        3D可视化层 (Visualization)        │
│  ┌─────────────┐  ┌─────────────┐      │
│  │ Three.js    │  │ 3d-force-   │      │
│  │ 渲染引擎    │  │ graph物理   │      │
│  └─────────────┘  └─────────────┘      │
└─────────────────────────────────────────┘
┌─────────────────────────────────────────┐
```

### 3.3 数据流保持
- **历史数据**：继续使用 `/api/v1/causal/history`
- **实时更新**：继续使用Socket.IO事件
- **用户交互**：保持现有表单和抽屉逻辑
- **向后兼容**：API接口保持不变

## 四、渐进式实验计划

### 4.1 实验原则
1. **最小可行**：先实现核心功能，再逐步完善
2. **风险控制**：在test.py中验证技术可行性
3. **渐进迁移**：验证成功后逐步替换现有组件
4. **并行运行**：保持2D/3D并行，确保稳定性

### 4.2 实验阶段

#### 阶段一：基础技术验证（第1周）
**实验1：基础3D图搭建**
- 目标：验证Three.js + 3d-force-graph基础集成
- 产出：基础3D场景HTML文件
- 验证：3D渲染、基础交互

**实验2：因果数据适配**
- 目标：验证现有数据到3D图的转换
- 产出：数据适配器Python/JS代码
- 验证：颜色/大小映射、边连接

#### 阶段二：性能基准测试（第2周）
**实验3：性能基准测试**
- 目标：测试不同规模节点的性能
- 产出：性能测试报告
- 验证：FPS、内存使用、加载时间

**实验4：交互功能原型**
- 目标：实现基本交互功能
- 产出：交互原型代码
- 验证：节点点击、相机控制、搜索高亮

#### 阶段三：高级功能实验（第3-4周）
**实验5：相机巡航动画**
- 目标：实现沿因果链的相机移动
- 技术：Tween.js集成
- 验证：平滑动画、路径规划

**实验6：视觉效果增强**
- 目标：添加基础视觉效果
- 技术：环境光、辉光效果
- 验证：视觉提升、性能影响

### 4.3 实验验证标准

#### 技术可行性：
1. ✅ 3D渲染正常显示
2. ✅ 数据加载无错误
3. ✅ 交互响应时间<100ms
4. ✅ 浏览器兼容性通过

#### 性能基准：
1. **FPS**：>30fps（100节点），>15fps（1000节点）
2. **内存**：<500MB（1000节点）
3. **加载时间**：<3秒（100节点），<10秒（1000节点）

#### 功能完整性：
1. **数据映射**：正确显示颜色、大小、标签
2. **交互功能**：点击显示详情，相机可控制
3. **实时更新**：能动态添加/删除节点

## 七、风险管理

### 7.1 技术风险
1. **Three.js学习曲线**
   - 缓解：使用成熟的3d-force-graph库
   - 缓解：分阶段学习，先基础后高级

2. **性能问题**
   - 缓解：渐进式优化，设置节点上限
   - 缓解：实现LOD、视锥体剔除

3. **浏览器兼容性**
   - 缓解：检测WebGL支持，提供降级方案
   - 缓解：使用较新但稳定的Three.js版本

### 7.2 项目风险
1. **时间超支**
   - 缓解：明确阶段目标，定期评估进度
   - 缓解：优先核心功能，高级特性后续迭代

2. **功能丢失**
   - 缓解：保持API兼容，并行运行验证
   - 缓解：详细测试计划，确保功能完整

3. **用户接受度**
   - 缓解：早期原型获取用户反馈
   - 缓解：提供2D/3D切换选项

### 7.3 回滚方案
1. **代码保留**：保留Vis-Network版本代码
2. **配置切换**：通过环境变量切换2D/3D模式
3. **快速回滚**：准备一键回滚脚本
4. **数据兼容**：确保数据格式前后兼容
