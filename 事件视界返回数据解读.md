# 事件视界（Event Horizon）—— 返回数据解读

## 核心概念

**点击（观察）一个事件节点时，系统返回三层信息：**

```
┌─────────────────────────────────────────────┐
│  data: 焦点节点的全息信息                      │
│  ├─ 节点身份（serial_id, node_id）            │
│  ├─ 叙事内容（event_tuple）                   │
│  ├─ 元数据（block_tag, action_tag, owner_id） │
│  ├─ 前事件（parent_ids, preview_id）          │
│  └─ 后事件（next_ids）                        │
├─────────────────────────────────────────────┤
│  event_horizon: 视界内所有节点的 ID 速查列表    │
│  event_horizon_details: 视界内节点概要详情      │
│  ├─ 焦点节点（distance=0.0，含完整因果链）     │
│  └─ 非焦点节点（distance>0，语义"余光"节点）   │
├─────────────────────────────────────────────┤
│  updated_count: 权重重分配影响的节点数          │
│  actor_id/owner_id: 本次操作的身份上下文        │
└─────────────────────────────────────────────┘
```

> **形象类比**：聚光灯打在"焦点节点"上，你能看清它的全貌和前后事件脉络；而视界内的其他节点就像用"余光"扫到的，只看到它们的概要信息。

## JSON 结构逐字段注释

```jsonc
{
    // ==========================================================
    // 【顶层】API 状态信息
    // ==========================================================
    "status": "success",
    // "success" | "error" — 请求是否成功

    "message": "节点 E节点：中国传统书法中的魏碑笔法解析 权重已提升到60%（大股东模式）",
    // 人类可读的操作结果描述


    // ==========================================================
    // 【核心】data — 焦点节点（大股东）的全息信息
    // 这是你"盯着看"的那个事件，权重被提升到 60%
    // ==========================================================
    "data": {

        // --- 节点身份 ---
        "serial_id": 485,
        // 物理序号：该事件在数据库中的唯一数字 ID，用于精确定位和关联查询

        "node_id": "E节点：中国传统书法中的魏碑笔法解析",
        // 意志标识：用户手动输入的事件标题，语义化的人类可读名称

        // --- 节点叙事 ---
        "event_tuple": "魏碑作为北朝碑刻书法的统称…",
        // 事件二元组描述：该事件的完整叙事内容（可能是原文、摘要或地宫恢复的全量内容）
        // 注意：如果节点被代谢提炼过，系统会自动从地宫恢复原始完整内容

        // --- 节点状态 ---
        "survival_weight": 0.6,
        // 生存权重：焦点节点被提升到 60%（大股东模式），表示当前观察者的意志集中于此
        // 取值范围 0~1，越高代表越"被关注"

        // --- 元数据标签 ---
        "block_tag": "因",
        // 因缘标签："因"（原因）| "相"（未经确认的可能性结果预估）| "果"（结果）
        // 因果推演中的角色定位

        "action_tag": "贞",
        // 动作标签："贞"（首贞/初始占卜）| "又贞"（继续观察或持续投入）| "对贞"（验证/结果占卜）
        // 类似甲骨文占卜的仪式化记录方式

        "full_image_url": "",
        // 全息图片 URL：该事件附带的视觉证据（截图/照片），空字符串表示无图片

        "owner_id": "test",
        // 事件拥有者 ID：该事件属于哪个数据库/租户。
        // 事件视界扫描只能看到同 owner 的事件，防止跨库污染

        // --- 因果链：前事件 ---
        "parent_ids": [],
        // 父事件标题列表：该事件的前驱节点（多因多果模型，支持多个父节点）
        // 空数组 [] 表示首贞（该事件是因果链的起点，无前事件）

        "preview_id": 0,
        // 前事件 ID：第一个父事件的 serial_id
        // 0 表示无父事件（首贞）。⚠ 多父场景下仅返回第一个父节点

        // --- 因果链：后事件 ---
        "next_ids": [486],
        // 后事件 serial_id 列表：该事件的所有直接子节点
        // [486] 表示 serial_id=486 的节点紧随该事件之后
        // 空数组 [] 表示末端节点（因果链的终点，无后事件）
    },


    // ==========================================================
    // 【操作上下文】本次点击的身份信息
    // ==========================================================
    "updated_count": 28,
    // 权重重分配影响范围：本次"大股东"操作共重新计算了 28 个节点的权重（该 owner 下的全部节点）

    "actor_id": "user2",
    // 观察者 ID：执行点击操作的用户。权重更新只影响该用户的个性化视角

    "owner_id": "222302526",
    // 请求中携带的 owner_id（用于权重分配）。⚠ 可能与焦点节点自身的 owner_id 不同
    // 但事件视界扫描使用的是焦点节点自身的 owner_id（本例中为 "test"），防止跨 owner 污染


    // ==========================================================
    // 【视界】event_horizon — 视界内节点 ID 速查列表
    // 按语义距离由近到远排列，方便程序快速遍历
    // ==========================================================
    "event_horizon": [
        "E节点：中国传统书法中的魏碑笔法解析",   // distance=0.0 — 焦点节点自身
        "B节点：科技浪潮下的创造重塑",            // distance=22.24 — 语义相近
        "F节点：书法和写字的区别"                 // distance=26.21 — 仍在视界半径内
    ],


    // ==========================================================
    // 【视界详情】event_horizon_details — 每个视界内节点的概要
    // "余光"扫描结果：非焦点节点只返回 ID+叙事+距离+因果链
    // 因果链（parent_ids/next_ids）是否返回由 .env 中 HORIZON_DETAIL 控制
    // ==========================================================
    "event_horizon_details": [

        // --- 第 1 个：焦点节点自身 ---
        {
            "serial_id": 485,
            // 物理序号

            "node_id": "E节点：中国传统书法中的魏碑笔法解析",
            // 节点标题

            "event_tuple": "魏碑作为北朝碑刻书法的统称…",
            // 完整叙事内容

            "distance": 0.0,
            // 语义距离：与焦点节点的余弦距离 × 100
            // 0.0 = 完全一致（自身），数值越大语义越远
            // 距离公式：D = (1 - 余弦相似度) × 100

            "parent_ids": [],
            // 父事件标题列表（仅在 HORIZON_DETAIL=1 时返回，否则省略）

            "next_ids": [486]
            // 子事件 serial_id 列表（仅在 HORIZON_DETAIL=1 时返回，否则省略）
            // 当前输出中包含因果链是因为 HORIZON_DETAIL=1
        },

        // --- 第 2 个：语义相近的非焦点节点 ---
        {
            "serial_id": 481,
            "node_id": "B节点：科技浪潮下的创造重塑",
            "event_tuple": "随着算法与大语言模型的深度演进…",
            "distance": 22.24307010321094,
            // 距离 22.24 ≤ MAX_EYES(30)，在视界半径内
            // 该节点与焦点节点在语义空间中相近（都涉及文化/艺术/科技话题）
            // 🔴 HORIZON_DETAIL=0 时不返回 parent_ids 和 next_ids（节省 Token）
            "parent_ids": [],
            "next_ids": []
        },

        // --- 第 3 个：焦点节点的后事件（子节点） ---
        {
            "serial_id": 486,
            "node_id": "F节点：书法和写字的区别",
            "event_tuple": "书法是一种艺术，写字是一种技能",
            "distance": 26.20966147471604,
            // 距离 26.21 ≤ MAX_EYES(30)，仍在视界内
            // 该节点是焦点节点的直接子事件（因果链上的后事件）
            "parent_ids": [
                "E节点：中国传统书法中的魏碑笔法解析"
            ],
            // parent_ids 显示它的前事件正是焦点节点 — 因果脉络清晰可见
            "next_ids": []
            // 该节点无子事件，是当前因果链的末端
        }
    ]
}
```

## 三层信息的使用场景

| 层 | 字段 | LLM Agent 用法 |
|----|------|---------------|
| **焦点全息** | `data.*` | "看"大股东节点的完整内容，理解当前聚焦的事件 |
| **前后脉络** | `parent_ids` / `next_ids` | 沿因果链深入探索，向前追溯前事件或向后追寻后事件 |
| **余光扫视** | `event_horizon_details` | 一次性扫到语义相近的事件，发现跨因果链的隐藏关联 |

---

## 环境变量控制

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `MAX_EYES=30` | 30 | 视界半径。距离 D ≤ 该值的节点进入视界。越小越聚焦，越大越广角 |
| `HORIZON_DETAIL=0` | 0 | 0=非焦点节点不返回 parent_ids/next_ids（节省 Token）；1=返回完整因果链 |

---

## 技术细节

- **距离计算**：`D = (1 - 余弦相似度) × 100`，使用 PostgreSQL pgvector 的 `<=>` 算子
- **owner 隔离**：视界扫描使用焦点节点自身的 `owner_id`，不信任请求参数中的 owner_id，防止跨库事件污染
- **地宫恢复**：如果焦点节点被代谢提炼，系统在点击时自动从 `ains_archive_necropolis` 恢复完整内容
- **权重重分配**（大股东模式）：焦点节点固定 60%，同 owner 下其余节点按原权重比例分配 40%
