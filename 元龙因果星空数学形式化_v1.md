## 一、事件视界方程（Event Horizon Equation）

### 1.1 语义距离

节点 $i$ 与节点 $j$ 的语义向量分别为 $\mathbf{v}_i, \mathbf{v}_j \in \mathbb{R}^{768}$（由 nomic-embed-text 模型编码），
定义余弦相似度为：

$$\text{sim}(\mathbf{v}_i, \mathbf{v}_j) = \frac{\mathbf{v}_i \cdot \mathbf{v}_j}{\|\mathbf{v}_i\| \|\mathbf{v}_j\|} \in [-1, 1]$$

映射为语义距离：

$$\boxed{D_{ij} = (1 - \text{sim}(\mathbf{v}_i, \mathbf{v}_j)) \times 100}$$

其中：
- $D_{ij} = 0$ 当且仅当 $\mathbf{v}_i = \mathbf{v}_j$（完全同一）
- $D_{ij} = 200$ 当且仅当 $\mathbf{v}_i = -\mathbf{v}_j$（完全相反）
- 典型值：$\text{sim}=0.7$ 时 $D=30$

### 1.2 视界判定

设聚焦节点为 $p$，系统常量 $\text{MAX\_EYES} = 30$，则节点 $q$ 是否在 $p$ 的视界内由下式判定：

$$\boxed{q \in \mathcal{H}(p) \iff D_{pq} \le \text{MAX\_EYES}}$$

其中 $\mathcal{H}(p)$ 表示节点 $p$ 的事件视界（Event Horizon），
即语义空间中以 $p$ 为球心、$\text{MAX\_EYES}$ 为半径的闭球内的所有节点集合。

**代码实现**（`core/database.py:get_event_horizon`）：

```sql
SELECT n.node_id, n.serial_id, n.event_tuple,
       (n.semantic_vector <=> (
           SELECT semantic_vector FROM ains_active_nodes WHERE node_id = %s
       )) * 100 AS distance
FROM ains_active_nodes n
WHERE (n.semantic_vector <=> (
    SELECT semantic_vector FROM ains_active_nodes WHERE node_id = %s
)) * 100 <= %s
ORDER BY distance ASC
```

其中 `<=>` 是 pgvector 的余弦距离运算符，定义为 $1 - \text{sim}(\mathbf{v}_i, \mathbf{v}_j)$。

### 1.3 视界分层

视界内的节点按距离升序排列，形成**语义引力梯度**：

$$p^{(0)} = p,\; D_{pp} = 0 \quad\text{(焦点节点)}$$
$$p^{(1)},\;p^{(2)},\;\ldots,\;p^{(k)}\quad\text{满足}\;0 < D_{p,p^{(1)}} \le D_{p,p^{(2)}} \le \cdots \le \text{MAX\_EYES}$$

距离越小，语义关联越紧密（类似于广义相对论中，越靠近引力中心，时空弯曲越大）。

## 二、权重分配方程（Weight Distribution Equation）

### 2.1 大股东权重视界坍缩

当 LLM Agent 或人类点击节点 $p$ 时，该节点成为**大股东（Boss Node）**。
系统对其所在语义空间执行**意志坍缩**，权重重新分配：

$$\boxed{W_{boss} = 0.60}$$

$$\boxed{W_{other,i} = \frac{W_{other,i}^{(old)}}{\sum_{j \neq boss} W_{other,j}^{(old)}} \times 0.40}$$

其中：
- $W_{boss}$：大股东节点固定占 60% 总权重
- $W_{other,i}$：任意非大股东节点按原权重比例分配剩余 40%
- 补充条件：若 $\sum_{j \neq boss} W_{other,j}^{(old)} = 0$，则所有非大股东节点平均分配 40%

**物理含义**：
> 观察者的意志（点击/检索）注入后，语义空间的权重分布坍缩。
> 被关注的节点获得绝对引力优势（60%），其他节点按各自历史贡献分剩余份额。
> ——这与量子力学中观察导致波函数坍缩在形式上同构。

### 2.2 个性化权重视界

系统支持多租户权重隔离。用户 $u$ 有权重表 `ains_user_weights`：

$$W_{u,i} = \text{get\_user\_weight}(u, \text{serial\_id}_i)$$

当用户 $u$ 点击节点 $p$ 时，仅更新 $u$ 的个性化权重表，不干扰其他用户的视界：

$$W_{u,boss} = 0.60$$
$$W_{u,other,i} = \frac{W_{u,other,i}^{(old)}}{\sum_{j \neq boss} W_{u,other,j}^{(old)}} \times 0.40$$

## 三、代谢引擎方程（Metabolism Engine Equation）

### 3.1 时间衰减函数（核心）

代谢引擎的核心是**指数衰减模型**，半衰期为 $\tau_{1/2}$：

$$\boxed{W(t) = 0.5^{\,t / \tau}}$$

其中：
- $t$：自事件创建以来经过的时间（纳秒精度）
- $\tau$：半衰期，默认 $\tau = 7 \text{ days} = 7 \times 24 \times 3600 \times 10^9 \text{ ns}$$

**代码实现**（`core/metabolism.py:calculate_precise_weight`）：

```python
def calculate_precise_weight(create_time_ns, half_life_days=7):
    now_ns = time.time_ns()
    diff_ns = max(0, now_ns - create_time_ns)
    half_life_ns = half_life_days * 24 * 3600 * 10**9
    exponent = diff_ns / half_life_ns
    weight_float = math.pow(0.5, exponent)
    return Decimal(str(weight_float))
```

### 3.2 功劳分数模型

每次滴答（Tick）中，引擎为每个非初创节点计算**功劳分数（Merit Score）**：

$$
M_i = \underbrace{W_{create}(t_{create,i})}_{\text{创建时间贡献}} + \underbrace{W_{access}(t_{access,i}) \times f_{decay}(t_{access,i})}_{\text{访问历史贡献}} + \underbrace{\epsilon_i}_{\text{唯一性微扰}}
$$

三项详解：

**① 创建时间贡献**：

$$W_{create}(t_{create,i}) = 0.5^{\, (t_{now} - t_{create,i}) / \tau_{create}}$$
其中 $\tau_{create} = 7 \text{ days}$

**② 访问历史贡献**：

$$W_{access}(t_{access,i}) = 0.5^{\, (t_{now} - t_{access,i}) / \tau_{access}}$$
其中 $\tau_{access} = 1 \text{ day}$（访问权重半衰期更短，记忆易逝）

**衰减因子**：

$$f_{decay}(\Delta t) = \begin{cases}
1.0, & \Delta t < 3600\text{s} \quad\text{(1小时内)}\\
0.8, & \Delta t < 86400\text{s} \quad\text{(1天内)}\\
0.5, & \Delta t < \text{EVENT\_LOSS}\quad\text{(超时时间内)}\\
0.1, & \Delta t \ge \text{EVENT\_LOSS}\quad\text{(超时)}
\end{cases}$$

其中 $\text{EVENT\_LOSS} = 3000\text{s}$

**③ 唯一性微扰**：

$$\epsilon_i = \frac{\text{hash}(\text{node\_id}_i \|\; t_{create,i}) \bmod 10000}{100000} \in [0, 0.1)$$

保证即使两个节点完全一样，功劳分数也不同（避免除零和排序歧义）。

### 3.3 代谢周期

代谢引擎以固定间隔 $\Delta_{tick}$ 循环执行，当前配置 $\Delta_{tick} = 60\text{s}$。

每次滴答执行以下操作：

1. **获取非初创节点**：$\mathcal{N} = \{i \mid \text{parent\_id}_i \neq \text{NULL} \lor \exists j : \text{parent\_id}_j = \text{node\_id}_i\}$
2. **计算每个节点的功劳分数** $M_i$
3. **识别大股东**：$boss = \arg\min_{i \in \mathcal{N}} \Delta t_{access,i}$ 且 $\Delta t_{access,i} < 300\text{s}$
4. **分配权重**：按方程(2.1)分配
5. **执行代谢动作**：

#### 代谢动作一：地宫封存（遗忘）

$$\text{if } W_i < 0.0001 \land \Delta t_{access,i} > \text{EVENT\_LOSS}:$$
$$\quad \text{event\_tuple}_i \gets \text{NULL}$$
$$\quad \text{full\_image\_url}_i \gets \text{NULL}$$
$$\quad \text{vision\_level}_i \gets 9$$

原始内容已提前封存至地宫表 `ains_archive_necropolis`。

#### 代谢动作二：内容提炼（压缩）

$$\text{if } \text{len}(\text{event\_tuple}_i) > \lfloor W_i \times \text{EVENT\_MAX} \rfloor:$$
$$\quad \text{event\_tuple}_i \gets \text{LLM\_summarize}(\text{event\_tuple}_i, \text{target\_len} = \lfloor W_i \times \text{EVENT\_MAX} \rfloor)$$

其中 $\text{EVENT\_MAX} = 1000000$ 字符。

**物理含义**：
> 权重 $W_i$ 既是引力质量，也是内存预算。
> 节点权重越大，分配的字符数越多；权重越小，内容被压缩得越短，直至遗忘。
> ——这与热力学第二定律（熵增、信息遗忘）在形式上同构。

## 四、存在主义检索方程（Existential Retrieval Equation）

### 4.1 定义

存在主义检索是**点击 → 坍缩 → 视界**的三元操作。

当 LLM Agent 或人类点击节点 $p$ 时：

$$\boxed{\text{Click}(p) \xrightarrow{\text{instant}} \text{Restore}(p) \land \text{Weight}(p \to 0.60) \land \text{Horizon}(p, \mathcal{H}(p))}$$

即一个点击行为同时触发三个动作：

1. **地宫恢复（Restore）**：如果 $p$ 已被提炼，从 `ains_archive_necropolis` 恢复完整内容
2. **权重坍缩（Weight Collapse）**：按方程(2.1)将 $p$ 设为 60% 大股东
3. **视界扫描（Horizon Scan）**：按方程(1.2)扫描 $p$ 的视界 $\mathcal{H}(p)$

### 4.2 LLM Agent 的存在主义认知框架

Agent 在 `search_causal_by_serial()` 中获取：

```
=== 大股东节点（权重60%）===
  事件: <node_id>
  叙述: <event_tuple>
  父链 (n): [...]
  子链 (m): [...]

=== 事件视界（MAX_EYES=30，共k个节点）===
  [距离1] <node_id_A>: <概要>
  [距离2] <node_id_B>: <概要>
  ...
```

Agent 只需"聚焦"大股东节点，用"余光"扫到视界内所有语义相关节点。
这被称为**存在主义检索**——既不是关键字匹配，也不是向量检索，
而是**以观测者意志为原点、语义引力为半径的存在论扫描**。

## 第六章 贞卜数据模型：贞-又贞-对贞的数学形式化

### 6.1 贞卜本体论

元龙因果体系的数据记录层采用"贞卜"模型，脱胎于殷商甲骨文的占卜术语，映射到现代因果推理的三种动作：

| 贞卜术语 | 殷商本义 | 系统映射 | 因果角色 |
|----------|----------|----------|----------|
| **贞**（Zhen） | 卜问/起始占卜 | 因果链的起始事件 | 初始因 |
| **又贞**（You Zhen） | 再次占卜/追加 | 因果链的中间事件 | 中间因/过渡因 |
| **对贞**（Dui Zhen） | 对贞/验证占卜 | 因果链的结果事件 | 果/验证 |

### 6.2 节点数据结构

每个因果事件节点 $n$ 的形式化定义：

$$
n = \langle ID_n, \text{event\_tuple}_n, \text{action\_tag}_n, \text{block\_tag}_n, \text{parent\_ids}_n, \mathbf{W}_n, \mathbf{v}_n, \text{meta}_n \rangle
$$

其中：

| 字段 | 类型 | 含义 | 数学符号 |
|------|------|------|----------|
| `node_id` | `VARCHAR(500)` | 事件唯一标识符（人工可读） | $ID_n$ |
| `serial_id` | `INTEGER` (AUTO) | 物理序列ID（机器主键） | $sid_n$ |
| `event_tuple` | `TEXT` | 事件二元组内容描述 | $E_n$ |
| `action_tag` | `VARCHAR(50)` | 动作标签：贞/又贞/对贞 | $A_n \in \{\text{贞},\text{又贞},\text{对贞}\}$ |
| `block_tag` | `VARCHAR(50)` | 因缘标签：因/相/果 | $B_n \in \{\text{因},\text{相},\text{果}\}$ |
| `parent_id` | `VARCHAR(500)` | 父事件ID（`|`分隔的多父） | $P_n \subseteq \mathcal{P}(ID)$ |
| `survival_weight` | `NUMERIC` | 生存权重 | $W_n \in [0,1]$ |
| `semantic_vector` | `vector(768)` | 语义嵌入向量 | $\mathbf{v}_n \in \mathbb{R}^{768}$ |
| `full_image_url` | `TEXT` | 全息图片URL | $I_n$ |
| `owner_id` | `VARCHAR(500)` | 事件拥有者 | $O_n$ |
| `last_accessed` | `TIMESTAMP` | 最后访问时间 | $t_{access,n}$ |
| `created_at` | `TIMESTAMP` | 创建时间 | $t_{create,n}$ |
| `vision_level` | `INTEGER` | 视觉层级 | $L_n \in \mathbb{N}$ |

**物理主键**：$(ID_n, O_n)$ —— 同一用户不可创建相同ID的两个事件。

**串行主键**：$sid_n$ —— 系统自动生成，用于高效索引和关联。

### 6.3 贞卜动作的数学定义

#### 6.3.1 贞（首贞）：因果链的根

**条件**：
$$P_n = \varnothing \iff \text{is\_first\_node}(n) = \text{True}$$

**强制约束**：
$$A_n = \text{"贞"},\quad B_n = \text{"因"}$$
$$W_n = 1.0$$

**语义向量**：
$$\mathbf{v}_n = \text{embed}(ID_n \text{。} E_n) \in \mathbb{R}^{768}$$

**数据库SQL等价**：
```sql
INSERT INTO ains_active_nodes 
(node_id, parent_id, block_tag, action_tag, event_tuple, survival_weight, owner_id, semantic_vector)
VALUES ($ID_n, NULL, '因', '贞', $E_n, 1.0, $O_n, $v_n);
```

**物理意义**：首贞对应因果链中的**根节点**——一个没有前提条件的初始事件。它是整个因果叙事的起点，类似殷商卜辞中的"某日，王占曰..."。

#### 6.3.2 又贞（中间事件）：因果链的节点

**条件**：
$$P_n \neq \varnothing,\quad A_n = \text{"又贞"}$$

**因缘标签约束**：
$$B_n \in \{\text{因},\,\text{相}\}$$

即"又贞"不能标记为"果"——它只是通往结果的一个中间站。

**多父继承机制**：父节点可以是多个事件（`|` 分隔），表示汇聚因果关系：

$$P_n = \{p_1, p_2, \ldots, p_k\},\quad k \ge 1$$

意即：该事件的成立依赖于所有父事件的同时成立。

**语义向量**：与首贞同样方式生成，但event_tuple由用户输入决定。

**物理意义**：又贞对应因果链中的**非根中间节点**，代表从因到果的过渡环节。其成立必须溯源于一个或多个父事件。

#### 6.3.3 对贞（结果事件）：因果链的果

**条件**：
$$P_n \neq \varnothing,\quad A_n = \text{"对贞"},\quad B_n = \text{"果"}$$

**物理意义**：对贞是认知过程的**验证性动作**，标记一条因果链的终点（"果"）。它代表事件推演的结论性判断。

### 6.4 贞卜动作的数学公理

#### 公理1：因果不可逆（Causal Irreversibility）

事件链严格沿时间方向推进，不允许循环：

$$\forall n \in \mathcal{N}:\; n \notin \text{descendants}(n)$$

由 `get_causal_chain` 中的循环检测（visited set）保证。

#### 公理2：历史不可覆盖（Historical Immutability）

一旦 $(ID_n, O_n)$ 被写入，该组合不可被覆盖。更新操作必须通过 `update` 接口（变更新ID或走更新流程）：

$$\text{INSERT} \Rightarrow \text{UNIQUE CONSTRAINT}(ID_n, O_n)$$

若尝试以已有 $(ID, O)$ 创建新节点，返回错误：
> "冲突：事件ID已存在。历史不可覆盖，请为新动作指定唯一ID。"

#### 公理3：因果溯源（Causal Traceability）

非首贞节点创建时，其所有父节点必须在数据库中已存在：

$$\forall p \in P_n:\; \text{db.get\_node\_by\_id}(p) \neq \text{NULL}$$

若任何父节点不存在，返回错误：
> "因果断裂：找不到父事件'...'，请先录入前贞。"

### 6.5 因缘标签与因果链语义

因缘标签 $\text{block\_tag}$ 的三元系统：

| 标签 | 语义 | 出现在 | 数学含义 |
|------|------|--------|----------|
| **因**（Cause） | 前置条件/动因 | 首贞 $\lor$ 又贞 | $\text{Cause}(n) = \bigwedge_{p \in P_n} \text{Effect}(p)$ |
| **相**（Phase） | 过渡状态/中间相 | 又贞 | $\text{Phase}(n) = \text{Intermediate}(\bigwedge P_n, \bigvee N_n)$ |
| **果**（Effect） | 结果/结论 | 对贞 | $\text{Effect}(n) \Rightarrow \bigvee_{c \in N_n} \text{Phase}(c)$ |

其中 $N_n$ 表示 $n$ 的后续节点（子节点）集合。

**因→相→果的推演链**：
$$\text{Cause}_1 \rightarrow \text{Phase}_1 \rightarrow \cdots \rightarrow \text{Phase}_k \rightarrow \text{Effect}$$

### 6.6 贞卜动作的数字信号编码

#### 6.6.1 动作-标签状态机

贞卜系统的状态转换可表示为有限自动机：

```
                    又贞[因/相]
                    ┌──────────┐
                    ↓          │
  ○ ──→ 贞[因] ──→ 又贞[因/相] ──→ 对贞[果]
 起始     首贞       中间环节       结果
```

#### 6.6.2 动作-标签对与因果角色的对应

$$
(A_n, B_n) \mapsto \text{causal\_role}(n) \in \{\text{ROOT},\ \text{INTERMEDIATE},\ \text{LEAF}\}
$$

映射规则：
$$
\text{causal\_role}(n) = \begin{cases}
\text{ROOT}, & P_n = \varnothing \land A_n = \text{"贞"} \land B_n = \text{"因"} \\[4pt]
\text{INTERMEDIATE}, & P_n \neq \varnothing \land A_n = \text{"又贞"} \land B_n \in \{\text{"因"},\text{"相"}\} \\[4pt]
\text{LEAF}, & P_n \neq \varnothing \land A_n = \text{"对贞"} \land B_n = \text{"果"}
\end{cases}
$$

### 6.7 因果链的递归数学定义

#### 6.7.1 因果链

一条因果链 $\mathcal{C}$ 是从一个根节点出发、沿父子关系可达的所有节点的有序集合：

$$
\mathcal{C}(r) = \{n \in \mathcal{N} \mid r \in \text{ancestors}(n)\},\quad \text{其中 } P_r = \varnothing
$$

其中 $\text{ancestors}(n)$ 定义为节点 $n$ 的所有祖先节点的集合：

$$
\text{ancestors}(n) = \begin{cases}
\{n\}, & P_n = \varnothing \\
\{n\} \cup \bigcup_{p \in P_n} \text{ancestors}(p), & P_n \neq \varnothing
\end{cases}
$$

#### 6.7.2 因果链骨架

`get_event_skeleton` 实现的骨架函数：

给定任意节点 $sid_k$，递归全向抓取连通图：
1. **上行递归**：从 $sid_k$ 沿 $parent\_id$ 链向上找所有祖先
2. **下行递归**：从所有祖先沿 $parent\_id$ 链向下找所有后代

$$
\text{skeleton}(sid_k) = \text{RECURSIVE\_CTE}\big(\{sid_k\},\ \text{UP}(\cdot),\ \text{DOWN}(\cdot)\big)
$$

递归CTE的SQL表达：

```sql
WITH RECURSIVE
ancestors AS (
    -- 起始点：当前节点
    SELECT node_id, parent_id, serial_id FROM ains_active_nodes WHERE serial_id = $sid_k
    UNION
    -- 上行：找父节点
    SELECT n.node_id, n.parent_id, n.serial_id FROM ains_active_nodes n
    JOIN ancestors a ON n.node_id = a.parent_id
),
all_related AS (
    SELECT * FROM ancestors
    UNION
    -- 下行：找子节点
    SELECT n.node_id, n.parent_id, n.serial_id FROM ains_active_nodes n
    JOIN all_related ar ON n.parent_id = ar.node_id
)
SELECT DISTINCT serial_id FROM all_related;
```

#### 6.7.3 层级树结构

骨架数据被组织为嵌套JSON树：

$$
\text{tree}(r) = \{
    \text{node\_data}(r),\ 
    \text{子事件列表} = [\text{tree}(c_1), \text{tree}(c_2), \ldots, \text{tree}(c_k)]
\}
$$

其中 $\{c_1, \ldots, c_k\} = \{n \in \mathcal{N} \mid r \in P_n\}$ 是 $r$ 的所有直接子节点。

**层级识别规则**：
- **根节点（首层）**：$\text{前事件ID列表} = []$ 或包含的地图中找不到父节点
- **子节点**：$\text{前事件ID列表} \neq []$ 且父节点在结果地图中

### 6.8 贞卜数据全生命周期

```
                    ┌─────────────────────────────────────┐
                    │           贞卜数据生命周期             │
                    └─────────────────────────────────────┘

   创建阶段                   代谢阶段                    遗忘阶段
  ┌──────────┐          ┌──────────────────┐          ┌──────────┐
  │ 贞(首贞)  │  ──→    │ 权重指数衰减      │  ──→    │ 地宫封存  │
  │ W=1.0    │          │ W(t)=0.5^{t/τ}   │          │ (遗忘)    │
  │ P=∅      │          │ content提炼       │          │          │
  └──────────┘          └──────────────────┘          └──────────┘
       ↓                       ↓                           ↓
  ┌──────────┐          ┌──────────────────┐          ┌──────────┐
  │ 又贞     │  ──→    │ 点击恢复权重      │  ──→    │ 地宫恢复  │
  │ P≠∅     │          │ W→0.60(大股东)    │          │ (回忆)    │
  │ 因/相    │          │ v→锐化            │          │          │
  └──────────┘          └──────────────────┘          └──────────┘
       ↓
  ┌──────────┐
  │ 对贞(果) │
  │ B=果     │
  └──────────┘
```

生命周期中的关键转换点：

1. **创建**（genesis）：节点以 $W=1.0$ 进入活跃表，同时获得初始语义向量 $\mathbf{v} = \text{embed}(ID \text{。} E)$
2. **代谢衰减**：$W$ 按指数衰减，当 $\text{len}(E) > \lfloor W \times \text{EVENT\_MAX} \rfloor$ 时触发内容提炼
3. **地宫封存**：原始内容存入 `ains_archive_necropolis`，提炼后的短内容留在 `ains_active_nodes`
4. **点击唤醒**：点击节点时，从地宫恢复完整内容和锐化向量
5. **地宫恢复**：恢复完整 $E$，并重新计算 $\mathbf{v} = \text{embed}(ID \text{。} E)$，使语义距离缩小、节点重回视界
