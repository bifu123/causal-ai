# 因果星空：事件视界与存在主义检索算法实现蓝图

## 1. 哲学设定：上帝视角 vs 观察者视角

在因果星空的设计理论中，存在两种截然不同但又相辅相成的视角：

*   **上帝视角（客观物理规律）**：
    *   由前端 3D 物理引擎和后端的全局代谢引擎（`metabolism.py`）维持。
    *   节点之间的语义相似度决定了它们在空间中的绝对距离（相似则聚，相异则散）。
    *   节点的全局权重随着时间的推移自然衰减，直至被所有人遗忘并封存入地宫。
*   **观察者视角（主观意志坍塌）**：
    *   由多租户权重表（`ains_user_weights`）和**事件视界（Event Horizon）**机制维持。
    *   当观察者（如张三）将意志锁定在某个节点（如“我想养猪”）时，宇宙为该观察者发生坍塌。
    *   只有与该意志相关的事件才会显现，无关的事件将在该观察者的视界中被“虚化湮灭”。

## 2. 核心概念：事件视界与 MAX_EYES

`MAX_EYES` 不是一个简单的节点数量硬截断，而是一个**“时空与语义的引力半径”**。

*   **精确语义距离公式**：
    利用 PostgreSQL 的 `pgvector` 插件计算余弦相似度，并将其映射为距离：
    `距离 D = (1 - 余弦相似度) * 100`
    *(例如：相似度 0.7 -> 距离 30)*
*   **视界判定**：
    *   **视界内 ($D \le MAX\_EYES$)**：节点与当前聚焦节点语义相近，或者属于直接的因果脉络。它们将被拉入视界，供 LLM 检索和前端高亮展示。
    *   **视界外 ($D > MAX\_EYES$)**：节点与当前意志毫不相干。它们将被排除在视界之外。

## 3. 落地执行计划

为了在不破坏“上帝视角”全局代谢规律的前提下，完美实现“观察者视角”的视界聚合，我们将采取以下轻量且优雅的改造方案：

### 3.1 环境变量配置
在 `.env` 文件中引入视界半径配置：
```env
MAX_EYES=30
```

### 3.2 后端接口改造 (`main.py` & `core/database.py`)
改造现有的节点聚焦接口（如 `search_causal_by_serial` 或点击节点的 API）：
1.  当用户或 Agent 聚焦某个节点时，除了返回该节点本身的数据和父子 ID 信息外，额外触发一次“视界扫描”。
2.  通过 SQL 瞬间计算全图活跃节点到该聚焦节点的语义距离 $D$。
3.  筛选出 $D \le MAX\_EYES$ 的节点，将这些**视界内节点**的完整内容（包括 `event_tuple` 等）打包进 API 返回结果的 `event_horizon` 字段中。

### 3.3 前端视觉呈现 (`static/js/3d_main.js`)
实现“视界湮灭”与“上帝视角”的无缝切换：
1.  **视界湮灭（观察者视角）**：
    *   前端接收到点击响应中的 `event_horizon` 列表。
    *   遍历 3D 场景中的所有节点：如果在视界列表中，则保持正常高亮显示；如果不在列表中，则执行**虚化湮灭**（例如将透明度降至 0.1 或直接隐藏）。
    *   视觉效果：张三点击“养猪”后，满屏的无关节点瞬间暗淡，只有“养猪”相关的节点在眼前聚合。
2.  **上帝视角恢复**：
    *   劫持前端现有的 `btn-fit-view`（视角复位按钮）点击事件。
    *   当点击该按钮时，除了执行原有的视角居中逻辑外，将所有被虚化/隐藏的节点恢复为正常显示状态（透明度 1.0），让用户重新回到客观的上帝视角。

### 3.4 LLM Agent 工具层 (`tools.py`)
更新 `search_causal_by_serial` 函数的逻辑与文档：
*   使其不仅用于触发点击，还能直接获取当前事件视界内的所有高价值节点内容。
*   **存在主义检索**：LLM Agent 在思考时，只需“看一眼”大股东节点，就能用“余光”扫到视界内的所有相关节点。这不仅实现了跨链的语义聚合，还通过 `MAX_EYES` 完美锁死了上下文 Token 的消耗上限。
